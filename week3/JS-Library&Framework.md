## ✅각 라이브러리/프레임워크의 주된 차이점은 무엇인가?

### 공통점 : SPA(Single Page Application)

 - SPA는 단일 페이지로 이루어진 애플리케이션이다. 기존의 서버 사이드 렌더링에 비해 배포가 쉽고, 모바일 앱과 같은 UX를 제공한다.

 - 기본적으로 모든 정적 리소스를 최초 한 번만 다운로드 받는다. 새 페이지 요청 시, 필요한 데이터만 전달 받아 페이지를 갱신한다. 필요한 부분만 갱신하므로 새로고침이 발생하지 않고 속도도 빠르다.

 - 위의 장점과 더불어 처음에 다운로드 크기가 크므로 초기 속도는 느린 편이다.

 - SPA개발을 할 수 있게 도와주는 대표적 라이브러리&프레임워크로 리액트, 뷰, 앵귤러, 스벨트를 꼽을 수 있다.

#### 점유율

- 점유율은 리액트가 압도적이며, 그 뒤를 뷰와 앵귤러가 뒤따르고 있다. 스벨트는 3개의 거대 프레임워크에 도전장을 내밀고 있는 형세이다.

 

 #### 1️⃣ 리액트 : 리액트는 페이스북이 개발했으며 2013년 출시 되었다. 공식 문서에도 나와있듯이 프레임워크가 아닌 라이브러리이다. 따라서 추가적으로 필요한 것이 있을 경우 별도로 라이브러리를 설치하여 사용하게된다. 
 
**👍 장점 :**
- 배우기 쉽고 사용하기 쉬움
- 컴포넌트 기반: 재사용 가능한 코드
- 성능 및 속도
- 대규모 커뮤니티
- 페이스북의 지원

**👎 단점 :**
- JSX 문법의 존재

 #### 2️⃣ 뷰 : 에반 유(Evan You)가 2014년 개발한 프레임워크이다. 라이브러리를 거의 고민할 필요가 없다. 상태관리 또한 vuex를 쓰게된다. 게다가 2020년 뷰3 의 등장과 함께 타입스크립트 지원이 강해지며 인기가 올라가고 있다.

**👍 장점 :**
- 성능 및 속도
- 컴포넌트 기반: 재사용 가능한 코드
- 배우기 쉽고 사용하기 쉬움
- 훌륭하고 직관적인 문서


**👎 단점 :**
- 리액트에 비해 비해 적은 리소스
- 과도한 유연성
 

 #### 3️⃣ 앵귤러 : 구글이 개발한 프레임워크로 2010년 출시되었다. 가장 먼저 타입스크립트를 도입하였고,  뷰 이상으로 모든 기능을 포함한 전체 프레임워크 세트를 제공하여 무겁다. 현재는 이전 버전을 AngularJS(Angular1)로 두고, Angular(Angular2+)로 계속 버전 업데이트가 이루어지고 있다.

**👍 장점 :**
- 구글의 지원
- 빠른 서버 성능
- MVC 아키텍처 구현
- 컴포넌트 기반: 재사용 가능한 코드
- 훌륭하고 직관적인 문서

**👎 단점 :**
- 복잡한 구조에 따른 가파른 러닝커브


 #### 4️⃣ 스벨트 : Rich Harris가 만들었고, 가장 늦은 2016년에 첫 배포가 이루어졌다. 놀랍게도 스벨트는 프레임워크도 아니고 라이브러리도 아닌 컴파일러이다. 빌드 시에 자체적으로 컴파일러가 돌며 번들파일에 코드를 담는다. 이 때 내 코드에서 사용하는 라이브러리의 부분만을 가져와 자체적으로 트리 쉐이킹을 한다.
 
**👍 장점 :**
- 가상 DOM 없음
- 진정한 반응성
- 배우기 쉽고 사용하기 쉬움
- 컴포넌트 기반: 재사용 가능한 코드

**👎 단점 :**
- 작은 커뮤니티
- 변수 이름 및 구문의 혼동가능성
- 대규모 앱에 적합하지 않음

#### 👉🏻요약

**인기도** : 리액트 > 뷰 > 앵귤러 > 스벨트
**성능** : 스벨트 > 리엑트 > 뷰 > 앵귤러
**러닝커브** : 스벨트 > 뷰 > 리액트 > 앵귤러
**커뮤니티와 리소스** : 리액트 > 앵귤러 > 뷰 > 스벨트

## ✅스타트업을 창업한다고 할 때, 무슨 라이브러리/프레임워크를 사용해야 할까?

- 위에서 정리한 프레임워크별 우위를 단순하게 정리하면 다음과 같다. 

- 속도 : 모두 빠름
- 단순함 : 스벨트, 뷰
- 인기도 : 리액트, 앵귤러

- 프로젝트의 리더라면, 우리가 의도한대로 사용자가 제품을 사용할 수 있는지를 고려해야한다. 따라서 다음의 요소들을 고려할 것이다.
   + 앱이 느려서 사용자가 이용할 수 있는지? 
   + 너무 복잡해서 버그가 생기지는 않을지? 
   + 유지보수가 간편한지? 
   + 앱을 빠르게 개발하고 출시하는데 러닝커브가 문제가 되는지? 
   + 커뮤니티의 규모가 얼마나 큰지? 

👉🏻 해당 요소들을 고려해서 기업의 제반사항에 알맞는 프레임 워크를 사용할 것 같다. 
👉🏻 리액트, 뷰만 겪어본 입장에서는 초기시장의 프로덕트라면 뷰를 택할 것 같고, 자본에 여유가 있다면 인력수급이 수월하고 큰 커뮤니티가 존재하는 리액트를 고르지 않을까 싶다 !! 
👉🏻 앵귤러는 정밀하고 고도화된 프로덕트에 이점이 있다고 해서, 만약 자본이 넉넉하다면 앵귤러를 고를 것 같다. 스벨트는 아직 상업 시장에서 존재감을 크게 드러내지 못해 어느정도 상업적으로 대중화가 된다면 고려할 것 같다. 


## ✅난 무엇을 위해 React 를 학습하는가?

- 리액트는 압도적인 점유율을 가지고 있고 큰 커뮤니티가 존재하기 때문에, 이는 **진입장벽이란 측면에서 큰 메리트**가 된다. 
- 실제로 내가 어떤 문제나 에러를 만났을때 이미 그 문제를 겪은 수많은 사람들의 도움을 받을 수 있었다. 
- 또한 시장에서 압도적 점유율을 가진 라이브러리이기 때문에, 동료를 찾기도 쉽고, **취업의 폭을 넓힐 수 있다**는 현실적인 장점도 있다.
- 리액트는 앵귤러의 가파른 러닝 커브, 스벨트의 부족한 대중성, 뷰의 과도한 유연함과 단순함 사이에서 중용의 미덕을 지키는 라이브러리라고 생각한다. 
- 리액트를 익히면 웹앱 뿐 만 아니라 리액트 네이티브를 통해 모바일 앱 또한 제작할 수 있다는 장점이 있다. (확장성)
- 이를 요약하자면 **기회비용**의 측면에서 리액트가 제일 우월하다고 생각하기 때문에 학습하는 것이다.
